# 🏃 ЗОЖник — Telegram-бот для формирования здоровых привычек

Учебный проект на Python, демонстрирующий разработку полнофункционального асинхронного Telegram-бота с ролевой моделью доступа, FSM, планировщиком задач, системой тикетов, геймификацией и AI-интеграциями.

---

## О проекте

**ЗОЖник** — Telegram-бот-помощник, который объединяет в одном интерфейсе:
- трекер воды;
- трекер физической активности;
- трекер сна;
- статистику и прогресс пользователя;
- систему обращений к администрации.

Главная учебная цель проекта: показать не только «работающие кнопки», но и **архитектурный подход** к построению бота, который устойчив к ошибкам, масштабируется по функционалу и имеет понятную модульную структуру.

---

## Что демонстрирует проект

1. **Асинхронную разработку на Python**
Используются `asyncio`, `aiosqlite`, `aiohttp`, что позволяет не блокировать обработку событий при работе с БД, сетью и Telegram API.

2. **Работу с Telegram Bot API**
Проект построен на `pyTelegramBotAPI` (режим `AsyncTeleBot`) с разделением обработчиков по типам входящих событий.

3. **Проектирование SQL-схемы под прикладную задачу**
Реализована связанная модель данных в SQLite: пользователи, трекеры, логи действий, тикеты, XP, упражнения, избранное.

4. **FSM-подход для многошаговых сценариев**
Настройки, рассылки, тикеты, админские действия и ввод произвольных параметров обрабатываются через состояния и контекст.

5. **Ролевую модель доступа (RBAC)**
Роли `User`, `Admin`, `Owner` с проверками доступа в критичных обработчиках.

6. **Фоновую обработку задач**
Планировщик (`utils/scheduler.py`) периодически обрабатывает напоминания, сбросы и сервисные задачи без участия пользователя.

7. **Устойчивость к нестабильной среде**
Есть обработка сетевых исключений, retry-логика polling, fallback-механизмы для медиа и мягкая деградация AI-функций.

8. **Интеграцию локального AI**
AI-анализ профиля реализован через Ollama (`gemma3:4b`) без зависимости от облачных API.

## Основной функционал

### Пользовательский контур

#### 1) Водный трекер

- Установка дневной цели по воде (готовые пресеты и ручной ввод).
- Быстрое добавление воды через inline-кнопки и custom-объём.
- Антиспам-ограничения на слишком частые добавления.
- Проверка безопасных лимитов на объём.
- Напоминания:
  - **Interval** — регулярные по заданному интервалу.
  - **Smart** — если пользователь долго не вносит воду.
- Недельный контур учёта и автоматический недельный сброс статистики.
- Поддержка таймзоны пользователя для корректного определения «сегодня».

#### 2) Трекер физической активности

- Управление дневной целью по количеству упражнений.
- Каталог упражнений по категориям и сложности.
- Режим избранного.
- Отметка выполнения упражнения с защитой от накрутки.
- Подсчёт статистики:
  - за день;
  - за неделю;
  - топ выполняемых упражнений.
- Напоминания об активности (интервальные и smart-режим).

#### 3) Трекер сна

- Настройка времени отбоя и подъёма.
- Фиксация начала и окончания сна.
- Вычисление длительности сессии сна.
- История сна и агрегаты за период.
- Напоминания:
  - перед временем сна;
  - вокруг времени подъёма.
- Логика «тихих часов» для подавления лишних уведомлений.

#### 4) Статистика и геймификация

- Профиль с агрегированной статистикой по всем трекерам.
- XP-система (начисление за действия).
- Уровни и рейтинг.
- Таблица лидеров.
- Сообщения о прогрессе и повышении уровня.

#### 5) AI-анализ профиля

- Сбор профиля пользователя на основе статистики воды/активности/сна.
- Запрос к локальной LLM через Ollama.
- Возврат персонализированного анализа в чат.
- Если Ollama недоступна — основной функционал продолжает работать без падения системы.

### Административный контур

- Доступ к админ-панели и административной статистике.
- Поиск пользователя по ID или username.
- Просмотр расширенного профиля пользователя.
- Бан/разбан пользователя.
- Ручная коррекция XP.
- Рассылки пользователям.
- Работа с тикетами (просмотр, ответ, обработка очереди).
- Управление упражнениями (добавление, редактирование, удаление, статистика).

### Контур владельца (Owner)

- Добавление администраторов.
- Снятие администраторских прав.
- Возврат прав администратора.
- Контроль административного состава без прямого редактирования БД.

## Система обращений (тикеты)

Подсистема обращений реализует двусторонний канал «пользователь ↔ администрация».

- Категории обращений: техподдержка и консультации.
- Создание обращения пользователем.
- Переписка в рамках тикета.
- Поддержка медиа-сообщений.
- Постраничный просмотр списков тикетов.
- Разделение статусов для пользователя и администратора.
- Блокировка тикета за администратором (lock) для исключения конфликтов при одновременной обработке.
- Периодическая очистка «зависших» lock-ов планировщиком.

## Архитектура проекта

### Точка входа

`main.py` отвечает за:
- инициализацию БД;
- запуск планировщика;
- регистрацию обработчиков сообщений/медиа/callback;
- запуск polling;
- безопасную остановку сервиса.

### Роутинг обработчиков

Для снижения связности и упрощения сопровождения логика разнесена по роутерам:

- `handlers/routers/message_router.py`
Маршрутизация обычных пользовательских текстовых сообщений.

- `handlers/routers/callback_router.py`
Маршрутизация callback-запросов с inline-клавиатур.

- `handlers/routers/fsm_router.py`
Маршрутизация входящих сообщений, когда у пользователя активен state.

Такое разделение делает точку входа компактной и позволяет изолированно развивать каждый поток.

### FSM

FSM реализована в `utils/fsm.py`:
- хранение состояния пользователя;
- хранение контекста текущего сценария;
- очистка/переходы состояний;
- многошаговые процессы (настройки, рассылки, тикеты, админские действия).

### Планировщик

`utils/scheduler.py`:
- запускается вместе с ботом;
- циклически проверяет необходимость напоминаний;
- отправляет уведомления по условиям;
- выполняет сервисные операции (weekly reset, cleanup lock).

### Слой данных

`database.py`:
- инициализация таблиц;
- CRUD-функции;
- агрегации статистики;
- операции по ролям, тикетам, упражнениям, sleep/water/activity и XP.

## Схема данных (SQLite)

Ключевые сущности:

- `users`
Пользователь, роль, timezone, даты регистрации/активности.

- `track_water`, `water_logs`
Настройки трекера воды и история добавлений.

- `track_activity`, `exercises`, `exercises_logs`, `exercises_user_favorites`
Цели активности, каталог упражнений, история выполнения, избранное.

- `track_sleep`, `sleep_logs`
Настройки сна и журнал сессий сна.

- `tickets`, `messages`
Тикеты и переписка по обращениям.

- `user_xp`, `xp_log`
Уровни, XP и журнал начислений.

Практические особенности:
- включены `FOREIGN KEY` связи;
- логовая модель в трекерах позволяет восстанавливать статистику;
- структура позволяет расширять функционал без критических миграций.

## Безопасность и устойчивость

- Проверки прав доступа в административных обработчиках.
- Блокировка действий забаненным пользователям.
- Ограничение параллельных отправок сообщений (`rate_limit_send`).
- Fallback для медиа в канале хранения (`media_storage`).
- Централизованные сообщения об ошибках в пользовательских потоках.
- Обработка исключений polling и автоматический повтор запуска.
- Очистка служебных зависших блокировок тикетов.

## Установка и запуск

### 1) Клонирование

```bash
git clone <repo_url>
cd bot
```

### 2) Виртуальное окружение

```bash
python -m venv .venv
```

Windows:

```bash
.venv\Scripts\activate
```

Linux/macOS:

```bash
source .venv/bin/activate
```

### 3) Установка зависимостей

```bash
pip install -r requirements.txt
```

### 4) Настройка `owners.json`

Пример:

```json
[6295868318]
```

`owners.json` определяет пользователей с ролью владельца (Owner), которым доступны операции управления администраторами.

### 5) Настройка `.env`

```env
TOKEN_BOT=your_telegram_bot_token
MEDIA_STORAGE_CHANNEL_ID=-1001234567890
```

Пояснение:
- `TOKEN_BOT` — токен Telegram-бота от BotFather.
- `MEDIA_STORAGE_CHANNEL_ID` — ID канала, куда бот складывает медиа для надёжной повторной отправки.

### 6) (Опционально) Запуск Ollama для AI-анализа

```bash
ollama pull gemma3:4b
ollama serve
```

### 7) Запуск проекта

```bash
python main.py
```

## Конфигурация и переключатели

В `config.py` используются ключевые параметры:

- `ai_censor_enabled` — включение AI-цензуры.
- `ai_analyzer_enabled` — включение AI-анализа профиля.
- `censorship_threshold` — порог токсичности для AI-проверки.
- `owners` / `owners_copy` — список владельцев из `owners.json`.

Также в `config.py` находится проверка ролей:
- `is_admin(...)`;
- `is_owner(...)`.
